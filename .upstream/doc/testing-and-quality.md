# Testing & Quality

Skia's test pyramid is built around four cooperating directories at the
repo root — `gm/`, `tests/`, `bench/`, and `fuzz/` — plus a single
unifying runner, `dm/`, that knows how to drive the first two against
every supported backend and produce a deterministic JSON manifest of
hashes and PNG images. Per-module suites
(`modules/skottie/tests/`, `modules/skparagraph/tests/`,
`modules/skshaper/tests/`, `modules/skunicode/tests/`,
`modules/skcms/tests/`, etc.) follow the same conventions.

| Directory | Purpose | Built into |
| --- | --- | --- |
| `gm/` | one `.cpp` per visual case → reference PNG | `dm` (renderer) |
| `tests/` | C++ unit tests using `Test.h` macros | `dm` (`--src tests`) |
| `bench/` | micro / mid-level benchmarks | `nanobench` |
| `fuzz/` | libFuzzer / AFL fuzz targets and seed corpora | `fuzz` binary |
| `dm/` | the cross-backend runner — the "diamond master" | standalone |

This separation is load-bearing: `gm/` is **the source of truth** for
visual regressions (Gold compares hashes on a per-pixel basis), while
`tests/` covers behaviour that doesn't lend itself to a single
reference image. Both compile into the same `dm` binary, which is then
the one tool a developer (or the CI bot) runs to validate a CL.

The kanvas project specifically depends on `gm/` outputs as ground
truth: `kanvas/src/test/resources/original-888/` carries 989 PNG
references rendered by upstream Skia from the corresponding GMs.

---

## DM — `dm/`

`dm` ("diamond master") is the universal runner. Its main entry point
is `dm/DM.cpp`, with the source/sink abstraction in `DMSrcSink.{h,cpp}`
and JSON output in `DMJsonWriter.{h,cpp}`.

The model is:

- **Sources** (`Src` subclasses): `GMSrc` (one per `.cpp` in `gm/`),
  `CodecSrc` (image decoders), `SKPSrc` / `MSKPSrc` (recorded
  pictures), `SVGSrc`, `SkottieSrc`, `BisectSrc`, etc.
- **Sinks** (`Sink` subclasses): `GPUSink` (per backend / per config —
  GL, Vulkan, Metal, D3D, Dawn), `CPUSink` (raster), `PDFSink`,
  `XPSSink`, `SKPSink`, `JSONSink`, `NullSink`.
- The cartesian product `Src × Sink` is the matrix DM walks. Each
  `(src, sink)` pair produces a PNG plus an MD5; the JSON manifest is
  uploaded to Skia Gold (`gold.skia.org`) which performs the diff.

DM also embeds the unit-test harness: with `--src tests` it iterates
every `DEF_TEST` / `DEF_GANESH_TEST` / `DEF_GRAPHITE_TEST` registered
through `tests/Test.h` and runs them inside the same process so that
GPU contexts can be shared.

Thread fan-out is via `SkTaskGroup`; tracing routes through
`tools/trace/ChromeTracingTracer` (writing a Chrome-tracing JSON) or
`SkDebugfTracer`. Hashing of pixel results uses `SkMD5` on
`HashAndEncode`-canonicalised pixels so colour-space differences don't
break determinism.

## GMs — `gm/`

A GM is a single `.cpp` file with a `DEF_SIMPLE_GM` (or `DEF_GM`) macro
that hands you an `SkCanvas` and a fixed-size canvas dimension. The
job of the file is to draw a deterministic image; everything else
(naming, registration, pixel comparison) is generated by the macro.

Examples in the directory range from primitives (`aaclip.cpp`,
`addarc.cpp`, `arcofzorro.cpp`) through filters
(`blurs.cpp`, `imagefilters_*.cpp`) to whole-feature integration
(`skbug_*.cpp` regression cases, `runtime_intrinsics.cpp` for SkSL).

GM output is the canonical reference — when porting to kanvas the
upstream `.cpp` *is* the spec, with the mechanically-extracted Javadoc
in `kanvas/src/generated/tests/org/skia/tests/` covering only the
`onDraw` body.

## Unit tests — `tests/`

`tests/Test.h` exposes the registration macros:

- `DEF_TEST(Name, reporter)` — pure-CPU test.
- `DEF_GANESH_TEST_FOR_RENDERING_CONTEXTS(Name, reporter, ctxInfo, ctxType)` —
  Ganesh test parameterised over context type.
- `DEF_GRAPHITE_TEST_FOR_*` — Graphite analogues.
- `DEF_CONDITIONAL_GANESH_TEST_FOR_RENDERING_CONTEXTS(…, condition)` — gated.

A `Reporter` accumulates failures (`REPORTER_ASSERT(r, cond, …)`,
`ERRORF(r, …)`). Tests live in flat `.cpp` files named after the
subject under test (`PathTest.cpp`, `BlendTest.cpp`, `PixmapTest.cpp`,
`GrTextureProxyTest.cpp`, …) — there are well over a thousand,
covering everything from `SkMatrix` to backend-specific shader caches.

## Benchmarks — `bench/`

`bench/Benchmark.h` is the GTest-style fixture for `nanobench`. Each
bench `.cpp` registers one or more `Benchmark` subclasses; nanobench
schedules them, warms them up, and reports per-loop time + standard
deviation. Categories include path operations
(`AAClipBench`, `BezierBench`, `BigPathBench`), blends
(`BlendmodeBench`), blurs (`BlurBench`, `BlurImageFilterBench`,
`BlurRectsBench`), and codec / decoder microbenchmarks
(`AndroidCodecBench`, `BitmapRegionDecoderBench`).

Wall-clock benchmarks of recorded scenes live separately in
`tools/skpbench/` (see [Developer Tools](developer-tools.md)).

## Fuzzers — `fuzz/`

`fuzz/Fuzz.{h,cpp}` plus `FuzzMain.cpp` provide a libFuzzer-compatible
entry point and a hand-written CLI. Per-target files
(`FuzzCanvas.cpp`, `FuzzPath.cpp`, `FuzzPathop.cpp`, `FuzzGradients.cpp`,
`FuzzRegionOp.cpp`, `FuzzEncoders.cpp`, `FuzzDDLThreading.cpp`,
`FuzzPrecompile.cpp`, `FuzzSkParagraph.cpp`, …) wrap a slice of the
public API in a deterministic byte-stream consumer. Seed corpora and
known-bad inputs live alongside the targets and are mirrored to
oss-fuzz.

Helper scaffolding — `FuzzCommon.{h,cpp}`, `FuzzCanvasHelpers.{h,cpp}`
— factors out random `SkPath` / `SkPaint` / `SkColor` synthesis so each
target stays small.

## Per-module tests

The same convention is repeated inside modules:
`modules/skottie/tests/` for [Skottie](skottie.md),
`modules/skparagraph/tests/` for [SkParagraph](skparagraph.md),
`modules/skshaper/tests/` for [SkShaper](skshaper.md),
`modules/skcms/tests/` for [skcms](skcms.md). They register through the
same `Test.h` macros and are picked up by the same `dm` binary.
