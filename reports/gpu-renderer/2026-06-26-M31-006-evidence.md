# KGPU-M31-006 — Bridge GPU Execution Evidence

## Scope

Execute `GPURecording` from `:kanvas` `Surface` on the offscreen WebGPU backend,
producing real RGBA pixels and writing them into the wrapped `SkSurface` for the
bridge production path. This is the prerequisite for M31-005 pixel parity.

## Status

**review** (PR #1887)

## Honest Scope

- **Supported**: `NormalizedDrawCommand.FillRect` with `SolidColor` material,
  `Identity` transform, `Root` layer scope, `WideOpen`/`DeviceRect` clip,
  `SrcOver` blend. Dispatched via `drawFullscreenPass` + scissor rect.
- **Refused**: everything else (FillRRect, FillPath, DrawTextRun, gradients,
  transforms, non-Root layers, complex clips). Emits stable `refuse:` diagnostic.
- **Production path**: `SkiaKanvasSurface.flush()` auto-renders to the wrapped
  `SkSurface` when Kanvas renderer is enabled. `flush()` itself still returns
  a `Frame` (side-effect rendering). On failure, emits `kanvas-render-failed`
  diagnostic (never silent no-op). Refused commands emit individual `refuse:`
  diagnostics via `emitBridgeDiagnostic`.
- **Default `flush()` remains record-only** on `Surface` directly. `renderToRgba()`
  and `flushAndRenderToSkSurface()` are opt-in for the `Surface`-level API.

## GPU Availability

WebGPU is only available in gradle JavaExec tasks (application JVM), not the
JUnit test JVM. All evidence produced via the tasks below.

## Known Limitations

- Only solid-fill rect dispatch supported; all other command types refused with
  stable diagnostics
- `Canvas.drawImage` lowered to solid-color FillRect (not an actual texture draw)
- Non-`SRC_OVER` blend modes refused
- GPU requires gradle JavaExec task; WebGPU unavailable in JUnit test JVM
- Default `Surface.flush()` (non-bridge) remains record-only; `renderToRgba()`
  is opt-in
